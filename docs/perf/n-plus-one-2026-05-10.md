# DB N+1 hunt — 2026-05-10

Phase J / Task 3.

## Method

Drizzle's pg client supports a `logger` option that prints every
emitted SQL statement. To count queries per request locally:

```ts
// apps/api/src/lib/db.ts (DEV-ONLY override)
import { drizzle } from 'drizzle-orm/node-postgres'

const db = drizzle(pool, {
  schema,
  logger: { logQuery: (q) => console.log('[sql]', q) },
})
```

Then `pnpm --filter @wistmail/api dev` and `grep -c '\[sql\]'` on
the request's stderr window. Threshold target per the plan: ≤ 5
queries for a 20-row response.

This logger was used during the audit to count queries on the
three list endpoints below; the override is **not** committed —
debug logging is removed before commit per Phase J guardrails.

## Endpoints audited

### `GET /api/v1/inbox/list?folder=inbox`

Implementation: `apps/api/src/services/feed.ts` → `FeedService.list()`.

Query count for a 50-row mixed feed: **5–7** depending on whether
chat conversations or attachments are present.

1. The unified feed query (`emails ⨝ mailboxes ⨝ users`, joined
   inside the same select).
2. `select id from mailboxes where userId = ?` — scoping check.
3. `select … from conversations ⨝ conversation_participants` for
   any chat rows in the page.
4. A single batched `WHERE conversation_id IN (…)` follow-up that
   pulls the *other* participants for each conversation.
5. The latest message per conversation via `DISTINCT ON
   (conversation_id)` — single round-trip even for 50
   conversations.
6. (Optional) Reactions / unread counters — already aggregated
   server-side, no per-row work.

No N+1; nothing changed. Comment trail in `feed.ts` already
documents the batching pattern.

### `GET /api/v1/projects`

Implementation: `apps/api/src/routes/projects.ts:29`.

Query count for a 20-row response: **2**.

1. The projects list (one column-projection select).
2. The batched `count(*) FILTER (WHERE status = 'done')` /
   `count(*)` per `projectId`, scoped by `inArray(projectIds)`.

No N+1. Note: the list query is a `db.select()` (every column),
including `description` (text). Description is needed by the
project-card UI so leaving as-is; if/when it's only shown on the
detail view we can drop it from the list.

### `GET /api/v1/admin/members`

Implementation: `apps/api/src/routes/admin.ts:175`.

Query count: **2**.

1. `select orgId from org_members where userId = ?` — find the
   caller's org.
2. `select … from org_members ⨝ users where orgId = ?` — every
   member with their joined user info.

No N+1.

## Quick wins applied

Both wins were spotted during the same pass and are written up in
detail in `api-response-times-2026-05-10.md`:

- **`/work/counters`**: 5 queries → 2 queries (single grouped
  `count(*) FILTER (WHERE …)` instead of 4 sibling counts).
- **`/docs`**: dropped the `body` column from the list projection.

## Observation

The codebase's services consistently use either a single windowed
select with joins or a `WHERE id IN (…)` batched follow-up. No
classic per-row N+1 was found in the three audited list endpoints.
The earlier query-count audits in `Phase H` and the comments
sprinkled through `feed.ts` and `chat.ts` already reflect this
discipline.
