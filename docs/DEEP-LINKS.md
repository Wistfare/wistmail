# Deep links — password reset

The mobile app ships with two entry points into `/auth/reset-password`:

## 1. Custom scheme (`wistmail://...`)

Works without any server-side setup. The user's mail client needs to
honour non-standard schemes, which iOS Mail and Gmail both do for
tapped URLs.

- **Android**: declared in `apps/mobile/android/app/src/main/AndroidManifest.xml`
  as an `intent-filter` with `<data android:scheme="wistmail" />`.
- **iOS**: declared in `apps/mobile/ios/Runner/Info.plist` under
  `CFBundleURLTypes`.

## 2. Universal / App Links (`https://wistfare.com/reset-password`)

Preferred — the link in the email looks like a normal web URL, and
tapping it opens the app directly when it's installed. Falls back to
the browser (where the existing web `/reset-password` page handles it)
when the app isn't installed, without the user noticing.

Requires two JSON files on the domain. Neither file is generated by
this repo; operators publish them out-of-band.

### Android — `https://wistfare.com/.well-known/assetlinks.json`

```json
[
  {
    "relation": ["delegate_permission/common.handle_all_urls"],
    "target": {
      "namespace": "android_app",
      "package_name": "com.wistfare.wistmail",
      "sha256_cert_fingerprints": [
        "<SHA256 of your release signing key, colon-separated>"
      ]
    }
  }
]
```

To get the fingerprint:

```sh
keytool -list -v -keystore /path/to/release.keystore \
  -alias <alias> | grep SHA256
```

### iOS — `https://wistfare.com/.well-known/apple-app-site-association`

```json
{
  "applinks": {
    "apps": [],
    "details": [
      {
        "appID": "<TEAMID>.com.wistfare.wistmail",
        "paths": ["/reset-password", "/reset-password?*"]
      }
    ]
  }
}
```

Served with `Content-Type: application/json`, no redirects. You can
sanity-check with Apple's validator:
<https://search.developer.apple.com/appsearch-validation-tool>

## Dart-side handling

`apps/mobile/lib/core/deep_links/deep_link_handler.dart` owns both the
cold-start link (the URL that launched the process) and the runtime
stream (tapping a link while the app is alive). Both paths flow into
`DeepLinkHandler._resolve(Uri)`, which currently understands a single
route: `/reset-password?token=…`. Adding another deep link target is a
new case in that method.

The custom scheme and the https App Link share the same `path` after
Flutter's Uri parsing (`/reset-password`), so the handler doesn't need
to branch on scheme.
